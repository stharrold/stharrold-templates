# initialize-repository (tactical notes)

Meta-skill that bootstraps a new repo with the full workflow system
(skills, docs, slash commands, pyproject, git setup). See
[`SKILL.md`](SKILL.md) for the canonical skill definition, full script
reference, Q&A flow, and component mapping.

This file holds only gotchas that cost someone >30 minutes to rediscover.

## Gotchas

### Source / target validation

- **"Source repository does not have workflow system"** means the
  source has <3/9 expected skills in `.claude/skills/`. If you see
  this, you're running against a non-template repo. Use
  `stharrold-templates` or another repo that has already been
  bootstrapped.
- **Target must not exist.** The script refuses to initialize into
  an existing directory to avoid clobbering. Either use a fresh path
  or delete the target first.
- **`git` is the only hard prerequisite.** `gh` is detected
  dynamically: if present and authenticated, the script offers to
  create a GitHub remote; if not, it skips that step and continues.
- **`gh auth status` must be green** before the GitHub-remote step
  runs. Cached credentials from a stale login produce confusing
  "unable to access remote" errors later. When in doubt, re-run
  `gh auth login`.

### Component copy rules (the `COPY_VERBATIM` vs `ADAPT_FILES` split)

Two internal constants decide what gets copied verbatim vs what gets
customized during the bootstrap. They are the hidden contract between
this skill and every other workflow skill:

- **`COPY_VERBATIM`**: `.claude/skills/`, `WORKFLOW.md`,
  `CONTRIBUTING.md`, `.gitignore`. These are tech-agnostic -- no
  per-repo customization. If you add a new skill that should also be
  copied verbatim, update this list AND the corresponding entry in
  `BUNDLES.md`.
- **`ADAPT_FILES`**: `README.md`, `CLAUDE.md`, `pyproject.toml`,
  `CHANGELOG.md`. These get templated with answers from the
  interactive Q&A (repo name, description, license, Python version).
  The adapt logic is in `initialize_repository.py::generate_*()` --
  one function per file.
- **`SKILL_NAMES`**: the 7 skills the meta-skill expects to find in
  the source (updated in #242: removed `agentdb-state-manager` and
  `workflow-orchestrator` which were deleted).

### Version and timestamp format

- **`TIMESTAMP_FORMAT = '%Y%m%dT%H%M%SZ'`** is compact ISO8601 with
  no colons or hyphens. Used consistently across worktree names,
  TODO file names, and the new-repo init log. Colons would require
  shell escaping on every path reference.
- **Initial `pyproject.toml` version is `0.1.0`**, not `1.0.0`.
  Bumping to 1.0.0 is an explicit decision after first release.

### Interactive Q&A behaviour

- **Default answers come from the source repo's own `pyproject.toml`**
  when present (Python version, license). Overriding one answer
  does not cascade -- you have to manually set each override.
- **The `repo_name` answer feeds into directory structure creation**;
  if it contains slashes or spaces the script rejects it with a
  clear error message.

### v8.9 notes

- **Per-directory CLAUDE.md stubs are no longer generated.** Earlier
  versions created `tests/CLAUDE.md`, `docs/CLAUDE.md`, etc. as
  autogenerated Tier-B stubs. That subsystem was removed (see
  `claude-md-hygiene` skill). New repos get exactly one root
  `CLAUDE.md` with the "What goes here" rubric at the top.
- **`gemini.ai/code` typo in earlier versions** of the generated
  root CLAUDE.md was a leftover from the Gemini -> Claude migration.
  Fixed in v8.9; if you see it in a bootstrapped repo, update the
  string manually (it's one line).
- **Slash commands use the `sN` prefix** (s1-worktree .. s4-backmerge),
  not `v7x1_N`. The meta-skill creates the files under
  `.claude/commands/workflow/`. If an older consumer still uses
  `v7x1_N`, it will be repaired on the next `--bundle git` re-apply.
